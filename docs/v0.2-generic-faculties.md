# AOP v0.2 sketch — making the neutrality claim true

**Status:** Design sketch / draft for discussion
**Date:** 2026-06-15
**Supersedes positioning in:** `rationale.md` "the blocks port near-verbatim from SonderEventCore"

## The problem this fixes

The v0.1 schema was lifted *directly* from Sonder's `SonderEventCore`. That made
it fast to ship and honest about its origin — but it means the supposedly neutral
faculty blocks are, field for field, Sonder's internal subsystems wearing a
standard's name. An adversarial read makes the charge stick:

| Field in v0.1 | What it actually is |
|---|---|
| `reasoning.neurotypes`, `reasoning.osi`, `reasoning.consensus`, `reasoning.rounds` | Parliament's multi-model debate engine |
| `governance.l1_pass` / `l2_pass` / `l3_pass`, `tier`="L0+L1", `circuit_state` | Lattice's three-tier gate engine |
| `memory.dream_cycle` | Engram's consolidation cycle |
| `capabilities.resolution` (LOD enum) | ACR's level-of-detail mounting |
| `intent.step_trace_id` | An internal StepTrace id |

A runtime that isn't built on Parliament/Lattice/Engram/ACR has no `osi`, no
`l1/l2/l3`, no `dream_cycle`. To emit a `full` event it would have to invent
values for fields that have no referent in its system. So in practice an outside
runtime emits `minimal` (a generic event header any logger already produces) or
nothing — and the "any runtime can emit a conformant event" claim is false.

**The rule that makes it true:** the normative fields must be derivable from the
*domain* (what any agent does), never from one implementation's architecture.
Implementation-specific richness is welcome — but it lives under
`metadata.<impl>`, not in the spec.

## The inversion

Today: generic concept is implicit, Sonder specifics are the schema.
v0.2: generic concept **is** the schema; Sonder specifics demote to `metadata.sonder`.

Each block keeps a small set of fields that *any* agent runtime can populate,
plus an open `metadata` escape hatch for impl-specific detail.

### `reasoning` — how a decision was reached

```jsonc
"reasoning": {
  // GENERIC — any runtime with a reasoning/planning step can fill these
  "strategy":   "single-model" | "multi-model" | "tool-assisted" | "other",
  "models":     ["string"],        // model id(s) involved
  "rounds":     1,                 // how many deliberation iterations
  "agreement":  0.0,               // 0..1 — degree of convergence, if measured
  "dissent":    ["string"],        // unresolved objections, if any
  "summary":    "string",          // human-readable basis for the decision
  "metadata":   {}                 // impl-specific: { sonder: { neurotypes, osi, consensus } }
}
```
- `neurotypes`, `osi`, Parliament's exact `consensus` object → `metadata.sonder`.
- A LangGraph shop sets `strategy:"single-model"`, `models:["gpt-x"]`,
  `summary:"..."` and is conformant. A Parliament emitter fills the generics
  *and* stashes its debate detail in `metadata`.

### `governance` — what policy permitted/blocked the action

```jsonc
"governance": {
  // GENERIC — any runtime with a policy/permission check can fill these
  "decision":   "allow" | "deny" | "needs_approval",
  "policy_id":  "string",          // identifier of the policy/contract evaluated
  "checks":     [                  // ordered list of rule evaluations
    { "rule": "string", "passed": true, "evidence": "string" }
  ],
  "violations": ["string"],
  "approval":   { "required": false, "granted": null, "approver": null },
  "metadata":   {}                 // impl-specific: { sonder: { l1_pass, l2_pass, l3_pass, tier, circuit_state } }
}
```
- Lattice's `l1/l2/l3`, `tier`, `circuit_state` → `metadata.sonder`. The generic
  `checks[]` array expresses the *same information* (a sequence of pass/fail rule
  evaluations with evidence) in a form any policy engine maps onto.
- A shop with a single `if-allowed` gate emits one `checks` entry and a
  `decision`. Lattice emits three checks + its tier detail in metadata.

### `memory` — what recall informed the step

```jsonc
"memory": {
  // GENERIC — any runtime with retrieval/recall can fill these
  "refs":       ["string"],        // ids/handles of recalled items
  "query":      "string",
  "confidence": 0.0,               // 0..1, if scored
  "metadata":   {}                 // impl-specific: { sonder: { dream_cycle } }
}
```
- `dream_cycle` → `metadata.sonder`. `refs`/`query`/`confidence` are what any
  RAG/retriever already has.

### `capabilities` — what the agent could do

```jsonc
"capabilities": {
  // GENERIC — any runtime that mounts tools can fill these
  "available":  ["string"],        // tool/capability ids the agent had
  "used":       ["string"],        // subset actually invoked
  "budget":     { "used": 0, "limit": 0 },   // optional resource accounting
  "metadata":   {}                 // impl-specific: { sonder: { resolution_lod } }
}
```
- ACR's LOD `resolution` enum → `metadata.sonder`. `available`/`used` is the
  generic tool-manifest any framework exposes.

### `prediction` — the agent's forecast of its own outcome

```jsonc
"prediction": {
  // GENERIC — any runtime that forecasts success can fill these
  "outcome":    "string",          // predicted result/label
  "confidence": 0.0,               // 0..1
  "model_id":   "string",
  "metadata":   {}                 // impl-specific: { sonder: { alpha, beta } }  (Beta-dist params)
}
```
- AWM's Beta `alpha`/`beta` → `metadata.sonder`. `outcome`+`confidence` is the
  generic forecast.

### `intent` — what the agent meant to do

```jsonc
"intent": {
  // GENERIC — any runtime with a plan/step can fill these
  "action":     "string",          // what the agent set out to do
  "skipped":    false,
  "skip_reason":"string",
  "metadata":   {}                 // impl-specific: { sonder: { step_trace_id, constraint_injected } }
}
```
- `step_trace_id`, `constraint_injected` → `metadata.sonder`.

## What stays the same

- **Identity envelope** (`aop_version`, `id`, `agent_id`, `task_id`,
  `timestamp`, `parent_id`) — already generic, no change.
- **`outcome` / `resources` / `paths`** — already generic, no change.
- **`trace_context`** — already optional/generic, no change.
- **Conformance tiers** — `minimal` / `governance` / `full` still hold, but now
  `full` is actually reachable by a non-Sonder runtime because every required
  field has a domain referent.

## Migration: Sonder keeps everything

This is non-destructive for Sonder. Its projection (`project()` /
`packages/core/src/aop.ts`) changes from "copy faculty fields up" to "map onto
generic fields, route the rest to `metadata.sonder`." Sonder loses no fidelity —
its rich fields are still emitted, just under `metadata` instead of at the
faculty top level. The existing `metadata.impl` demotion machinery (added in
`c35daf6`) is the same mechanism; this just widens what flows through it.

## The new litmus test

A reviewer should be able to read each generic block and answer "could a runtime
I've never heard of fill this from its own concepts?" with **yes**, without
knowing what Parliament/Lattice/Engram are. If a normative field requires knowing
a heybeaux product to populate, it's in the wrong place — push it to `metadata`.

## Open questions for v0.2

1. **`checks[]` vs `l1/l2/l3`** — is a flat ordered `checks` array expressive
   enough to losslessly carry Lattice's tiered results into `metadata`, or does
   governance need a generic `tiers` concept too? (Lean: flat array + metadata.)
2. **Should `metadata.<impl>` be schema-validated at all**, or fully opaque?
   (Lean: opaque — the whole point is the spec doesn't know about impls.)
3. **Versioning the move** — is this v0.2 (additive, with v0.1 faculty fields
   kept as deprecated aliases) or a clean v0.2 break? Since there are no external
   adopters yet, a clean break is cheaper than carrying aliases.
